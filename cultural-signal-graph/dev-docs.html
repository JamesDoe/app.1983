cultural-signal-graph

This document is a handoff-oriented overview of the current MVP. It explains what exists, where the validation boundary lives, how data moves through the system, and which parts are scaffolding versus production-relevant.

Current Scope

Implemented now

Validated ingestion from raw JSONL into parquet, UTC timestamp normalization, invalid-row logging, phrase extraction from comment parquet, reply graph reconstruction from comment parquet, cultural signal analysis from processed artifacts, and CLI entrypoints for all current stages.

Partially scaffolded

The core pipeline stages exist now. The remaining work is mostly about deepening the analysis layer and tightening cross-stage contracts rather than adding missing major components.

Design goal

Keep the system local, readable, and CLI-driven. There is no database, job queue, or dashboard in the current architecture.

Developer Workflow

This repository uses a src/ layout. In practice that means developers should install the package in editable mode before relying on module execution such as python -m csg.cli.ingest. 

python -m venv .venv
.venv\Scripts\Activate.ps1
python -m pip install -e .[dev]

After that, both of these are valid: 

python -m csg.cli.ingest
python -m csg.cli.extract_phrases
python -m csg.cli.build_graph
python -m csg.cli.analyze
csg-ingest --comments data/raw/comments.jsonl --threads data/raw/threads.jsonl
Running python -m csg.cli.ingest without an editable install or equivalent PYTHONPATH setup will usually fail in a standard src/ layout.

Key design choices

Area Decision Reason
Validation boundary src/csg/io/load_jsonl.py performs parse and model validation while streaming. Keeps bad data handling in one place and prevents CLI logic from mixing orchestration with row-level correctness.
Schema enforcement src/csg/models.py defines CommentRecord and ThreadRecord. Pydantic gives explicit required fields, type coercion, and a clean path for future schema evolution.
Output format Validated records and extracted phrase artifacts are written as parquet using Polars. Parquet is a better intermediate format for downstream analytics than JSONL and keeps file sizes manageable.
Phrase extraction src/csg/extract/phrases.py reads comment parquet through spaCy noun chunk extraction and emits phrase plus edge tables. This keeps phrase logic isolated and gives later graph stages stable artifacts to consume.
Reply graph reconstruction src/csg/graph/build_reply_graph.py resolves Reddit parent_id references into directed comment edges and thread metrics. This reconstructs thread structure from comment parquet without introducing a database or graph store.
Signal analysis src/csg/cli/analyze.py combines phrase, edge, reply, and comment artifacts into salience and trend outputs. This keeps the analysis stage artifact-driven and independent from raw-source parsing.
Error handling Malformed JSON and validation failures are skipped and appended to artifacts/errors.jsonl. Ingestion remains resilient on mixed-quality datasets while preserving debugging visibility.

Implementation Details

1. Models

CommentRecord and ThreadRecord define the expected ingest schema. Both normalize created_at to timezone-aware UTC datetime values using a shared helper.

The models are intentionally strict on required fields. A missing required field is treated as an invalid row, not silently defaulted.

2. Streaming JSONL loader

stream_jsonl() reads one line at a time. Each non-empty line is parsed as JSON and optionally validated against a supplied Pydantic model. This is the main ingestion primitive.

Failure modes:

  • Malformed JSON: logged with error_type = "malformed_json".
  • Schema or type failure: logged with error_type = "validation_error".
  • Valid rows: yielded to the caller as model instances.

3. CLI orchestration

python -m csg.cli.ingest is the top-level ingest command. It does not do row manipulation itself. It wires together source files, the streaming validator, and parquet serialization. 

python -m csg.cli.ingest \
  --comments data/raw/comments.jsonl \
  --threads data/raw/threads.jsonl \
  --comments-output data/processed/comments.parquet \
  --threads-output data/processed/threads.parquet \
  --errors artifacts/errors.jsonl

4. Parquet writer

The CLI currently materializes validated rows into memory before writing parquet. For the current MVP this is simple and acceptable, but it is the main scalability limitation in the ingest path.

If datasets grow substantially, the first change should be chunked parquet writing rather than broad architectural rewrites.

5. Phrase extraction stage

python -m csg.cli.extract_phrases reads data/processed/comments.parquet, loads a spaCy pipeline, extracts noun chunks from comment text, normalizes phrases, preserves local abbreviations such as hrbt, vb, odu, norfolk, and ghent, then writes two parquet outputs. 

python -m csg.cli.extract_phrases \
  --input data/processed/comments.parquet \
  --phrases-output data/processed/phrases.parquet \
  --edges-output data/processed/comment_phrase_edges.parquet

The outputs are:

  • phrases.parquet: one row per normalized phrase with ID, occurrence count, and first-seen timestamp.
  • comment_phrase_edges.parquet: many-to-many mapping between comment IDs and phrase IDs.

6. Reply graph stage

python -m csg.cli.build_graph reads data/processed/comments.parquet, resolves comment-to-comment edges from Reddit parent_id values, and computes both per-edge depth output and thread-level summary metrics. 

python -m csg.cli.build_graph \
  --input data/processed/comments.parquet \
  --edges-output data/processed/reply_edges.parquet \
  --metrics-output data/processed/thread_metrics.parquet

Internal per-comment metrics are computed during reconstruction:

  • verified_depth: recomputed depth from reconstructed parent-child links.
  • thread_root: root comment for the connected reply branch.
  • descendant_count: total descendants below the comment.
  • branch_size: size of the comment subtree including the comment itself.

The persisted outputs are:

  • reply_edges.parquet: parent_comment_id, child_comment_id, depth, thread_id.
  • thread_metrics.parquet: thread_id, comment_count, max_depth, avg_depth.

7. Cultural signal analysis stage

python -m csg.cli.analyze reads the processed phrase, comment-phrase edge, reply-edge, and comment parquet artifacts and writes CSV outputs for downstream review. 

python -m csg.cli.analyze \
  --phrases data/processed/phrases.parquet \
  --comment-phrase-edges data/processed/comment_phrase_edges.parquet \
  --reply-edges data/processed/reply_edges.parquet \
  --comments data/processed/comments.parquet \
  --output-dir artifacts

This stage currently produces:

  • phrase_frequency.csv: phrase-level occurrence counts derived from phrases.parquet.
  • phrase_cooccurrence.csv: pair counts for phrases that appear in the same comment.
  • contested_phrases.csv: phrases appearing in comments whose verified reply depth exceeds 5.
  • temporal_phrase_counts.csv: weekly phrase usage counts.
  • top_signals.csv: salience ranking based on frequency, comment score, and normalized reply depth.
The salience calculation clamps negative aggregate comment scores to zero before applying log(1 + total_comment_score). This prevents math-domain failures on phrases associated with net-negative scored comments.

Tests

The current tests cover ingestion, phrase extraction, reply graph reconstruction, and analysis:

  • Valid row parses and normalizes timestamp correctly.
  • Missing required field is rejected and logged.
  • Malformed JSON is rejected and logged.
  • CLI ingest helper writes parquet files for valid comment and thread inputs.
  • Phrase normalization preserves required regional abbreviations.
  • Phrase extraction builds phrase and edge tables from example comments.
  • Reply graph reconstruction resolves parent-child edges and thread metrics from example comments.
  • Analysis writes the expected artifact set and computes salience from artifact inputs.

These tests live in tests/test_ingestion.py, tests/test_phrase_extraction.py, tests/test_reply_graph.py, and tests/test_analysis.py.

Known Gaps

  • The parquet writer currently buffers all validated rows in memory before writing.
  • The phrase extractor assumes a spaCy pipeline with noun chunk support for best results. The blank-model fallback is still limited.
  • Reply graph metrics are computed internally, but only thread-level metrics are currently persisted because the task only requested edge and thread summary outputs.
  • The current analysis outputs are file-based summaries only; there is no persisted normalized analysis table for reuse across later stages.
  • Invalid row logging is append-only and does not rotate or partition by source file.
  • There is no schema versioning yet for processed outputs.

Recommended Next Steps

  1. Decide whether the analysis stage should also persist intermediate parquet tables in addition to the human-readable CSV outputs.
  2. Introduce a small repository-level data access layer so non-ingest modules do not care whether storage is parquet or something else later.
  3. Decide whether per-comment reply metrics should become a persisted artifact rather than remaining an internal computation.
  4. Add explicit integration tests against the real sample files under data/raw/ and the produced parquet outputs.
